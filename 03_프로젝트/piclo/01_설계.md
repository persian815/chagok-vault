## DB 스키마

```sql
-- 프로필 (Supabase Auth users 확장)
create table profiles (
  id uuid references auth.users on delete cascade primary key,
  email text,
  created_at timestamptz default now()
);
alter table profiles enable row level security;
create policy "own profile" on profiles
  for all using (auth.uid() = id);

-- Auth 트리거: 회원가입 시 profiles 자동 생성
create or replace function handle_new_user()
returns trigger as $$
begin
  insert into profiles (id, email) values (new.id, new.email);
  return new;
end;
$$ language plpgsql security definer;
create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function handle_new_user();

-- 사진 메타데이터
create table photos (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id) on delete cascade not null,
  storage_path text not null,         -- Supabase Storage 경로
  filename text not null,
  file_size int,
  width int,
  height int,
  taken_at timestamptz,               -- EXIF 날짜
  lat float,
  lng float,
  ai_category text,                   -- 'people' | 'food' | 'landscape' | 'document' | 'screenshot' | 'other'
  ai_tags text[],
  ai_confidence float,
  created_at timestamptz default now()
);
alter table photos enable row level security;
create policy "own photos" on photos
  for all using (auth.uid() = user_id);

-- 앨범
create table albums (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id) on delete cascade not null,
  name text not null,
  cover_photo_id uuid references photos(id),
  type text default 'manual',         -- 'auto_date' | 'auto_location' | 'auto_category' | 'manual'
  created_at timestamptz default now()
);
alter table albums enable row level security;
create policy "own albums" on albums
  for all using (auth.uid() = user_id);

-- 사진-앨범 N:M
create table photo_albums (
  photo_id uuid references photos(id) on delete cascade,
  album_id uuid references albums(id) on delete cascade,
  primary key (photo_id, album_id)
);
alter table photo_albums enable row level security;
create policy "own photo_albums" on photo_albums
  for all using (
    exists (
      select 1 from photos
      where photos.id = photo_id
        and photos.user_id = auth.uid()
    )
  );
```

## 핵심 설계 결정 (ADR)

### ADR-001: 파일 크기 제한 60MB
- **결정:** `next.config.mjs`에 `bodySizeLimit: '60mb'` 설정
- **이유:** Vercel 기본 4.5MB 제한으로 스마트폰 사진 업로드 실패했던 경험(삽질모음). 요구사항 파일당 최대 50MB이므로 60MB 설정.
- **구현:** `next.config.mjs` > `api: { bodyParser: { sizeLimit: '60mb' } }`

### ADR-002: EXIF 파싱은 브라우저 사이드
- **결정:** `exifr` 라이브러리로 클라이언트에서 GPS·날짜 추출 후 메타데이터만 서버 전송
- **이유:** 서버 부하 절감. Storage 업로드와 EXIF 파싱 병렬 처리 가능.
- **구현:** 업로드 전 `exifr.parse(file)` → `taken_at`, `lat`, `lng` 추출 → photo 레코드 저장 시 포함

### ADR-003: HEIC 브라우저 사이드 변환
- **결정:** `heic2any` 라이브러리로 클라이언트에서 JPEG 변환 후 업로드
- **이유:** Vercel 서버리스 메모리 제한으로 서버 이미지 변환 위험. 브라우저 변환이 안전하고 서버 비용 절감.
- **구현:** HEIC 파일 감지 시 변환 진행 UX 표시 후 업로드

### ADR-004: Gemini Vision 병렬 처리
- **결정:** `/api/classify`에서 최대 5장 병렬 처리, `Promise.allSettled` 사용
- **이유:** 100장 30초 이내 요구사항 충족. 실패 장만 에러, 전체 중단 방지.
- **구현:** `maxOutputTokens: 8192` 필수 (2048 설정 시 JSON 잘림 — 삽질모음)

### ADR-005: 모든 API 라우트에 인증 가드
- **결정:** `/api/*` 모든 라우트에 `auth.uid()` 확인 가드 적용
- **이유:** 게스트 가드 누락으로 비인증 접근 가능했던 경험(삽질모음). 사진 데이터는 특히 민감.
- **구현:**
  ```typescript
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  ```
