# 하루 만에 앱 완성 — 로그인부터 데이트 히스토리까지

로컬 저장소 없이, 로그인도 되고, 과거 기록도 편집할 수 있는 앱이 하루 만에 완성됐다.

---

## 오늘 한 것

오늘 하루는 꽤 바빴다. `dayOne` 프로젝트를 처음 배포한 날인데, 기능을 쌓고, 마무리 루틴을 돌리고, 또 기능을 쌓고를 반복했다.

### 이메일 로그인

처음에는 OAuth 소셜 로그인 이야기도 나왔는데 이메일+비밀번호로 결정했다. Supabase Auth에 `signInWithPassword()`를 붙이는 건 간단했다. 로그인 성공 후 `/`로 이동하고, Next.js의 `router.refresh()`를 같이 호출해서 서버 세션이 반영되도록 했다.

Next.js 16에서 인증 게이트를 처리하는 파일이 `middleware.ts`에서 `proxy.ts`로 바뀌었다. 빌드 결과를 보니 이런 경고가 떠 있었다:

```
⚠ The "middleware" file convention is deprecated. Please use "proxy" instead.
```

`src/middleware.ts`를 `src/proxy.ts`로 옮기고, 함수 이름도 `middleware`에서 `proxy`로 바꿨다. 처음엔 파일만 이름 바꾸고 함수 이름을 그대로 뒀더니 빌드가 이렇게 터졌다:

```
Proxy is missing expected function export name
```

함수 이름까지 같이 바꿔야 한다는 걸 두 번 만에 알았다.

### localStorage → Supabase 에피소드 저장

원래 에피소드(데이트 기록)는 브라우저의 `localStorage`에만 저장하고 있었다. 다른 기기에서 보면 데이터가 없고, 저장 공간 제한도 있었다. Supabase `episodes` 테이블에 저장하도록 전환했다.

문제가 하나 있었다. 기존 테이블이 `title`, `description`, `ical_data` 같은 범용 컬럼으로 만들어져 있었는데, 앱에서는 `ep`, `area`, `place`, `trip`, `start_time`, `end_time`, `memo`, `lat`, `lng` 같은 구체적인 필드가 필요했다. 마이그레이션 파일(`002_add_episode_columns.sql`)로 컬럼을 추가했다.

TypeScript 코드는 camelCase(`startTime`, `endTime`)를 쓰고, PostgreSQL은 snake_case(`start_time`, `end_time`)를 쓴다. API 레이어에 `dbToEpisode()` 변환 함수를 만들어서 양쪽을 맞췄다.

### 히스토리 수정 + ICS 재다운로드

히스토리 화면에서 에피소드 카드를 눌러 내용을 수정하고, 수정한 내용으로 캘린더 파일(`.ics`)을 다시 다운로드할 수 있게 됐다. 카드 아래 인라인으로 수정 폼이 펼쳐지는 UX로 구현했다.

### 134회 데이터 마이그레이션

ep 1번부터 134번까지의 과거 데이트 기록을 한 번에 넣을 SQL 시드 파일을 작성했다. PL/pgSQL `DO $` 블록으로 이메일 주소로 사용자를 찾아서 `user_id`를 붙여 `INSERT`하는 방식이다. `ON CONFLICT (user_id, ep) DO NOTHING`을 달아서 중복 실행해도 안전하다.

---

## 막혔던 부분

Next.js 16 proxy 함수 이름 문제가 제일 당황스러웠다. 파일 이름 바꾸는 것만으로 되겠거니 했는데 함수 이름도 같이 바꿔야 한다는 게 빌드 에러로만 알 수 있었다. 문서를 먼저 읽었으면 한 번에 끝냈을 것이다.

camelCase/snake_case 변환은 API 레이어에서만 처리하면 나머지 코드가 깔끔하다는 걸 다시 확인했다. 컴포넌트가 DB 컬럼명에 의존하면 나중에 바꾸기 훨씬 힘들다.
