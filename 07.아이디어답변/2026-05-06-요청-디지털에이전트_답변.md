# 대고객 디지털 에이전트 론칭 준비 가이드

> 요청 날짜: 2026-05-06  
> 답변 날짜: 2026-05-06

---

## 요청 요약

작년 임직원 대상으로 개발·오픈한 입원건강보험 AI Agent를 보완하여, 팩 건강보험상품에 대한 대고객 서비스로 10월 론칭을 목표로 한다. CBT에서 확인된 할루시네이션 30% 문제를 해결하고, 새 AI 거버넌스 심의를 통과하는 것이 핵심 과제다. 서비스 PM 관점에서 현재 계획에 추가로 필요한 사항을 정리한다.

---

## 답변

### 현재 계획 평가

정리한 5단계 로드맵(기획 재정의 → 착수보고 → 개발 → 테스트 → 오픈)은 뼈대가 잘 잡혀 있다. 다만 아래 항목들이 각 단계에 보완되어야 10월 론칭을 안정적으로 달성할 수 있다.

---

### 단계별 보완 사항

#### 1단계 | 기획 재정의 (5월 초)

**현재 계획에 추가할 것들:**

- **할루시네이션 30% 원인 분석 및 해결 방향 명문화**
  - 작년 CBT 결과를 기반으로 오답 유형을 분류한다 (모델 오류 / RAG 검색 실패 / 프롬프트 구조 문제 / 데이터 품질 문제).
  - Qwen 235B와 397B 모델의 사전 성능 검증(Bench test) 계획을 포함한다.
  - 허용 가능한 할루시네이션 임계치를 정의한다 (예: 대고객 서비스 기준 5% 이하).

- **대고객 전환 시 법적 검토 선착수**
  - 보험업법 상 AI를 통한 상품 안내·설명의 허용 범위
  - 금융소비자보호법 준수 방안 (적합성 원칙, 불완전판매 방지)
  - 개인정보보호법 (고객 입력 데이터 처리, 대화 로그 보관 기준)
  - **법무·컴플라이언스 검토는 5월 중에 완료**해야 6월 개발에 반영 가능하다.

- **성공 지표(KPI) 정의**
  - 할루시네이션율 (목표: 5% 이하)
  - 고객 만족도 / 상담 완결률
  - 상품 판매 연결률
  - 상담원 에스컬레이션 비율

- **고객 범위 및 진입 경로 확정**
  - 어떤 고객이 이 Agent를 만나는가? (앱 내 배너, 상품 상세 페이지, 채팅 버튼 등)
  - 팩 건강보험 가입을 고려 중인 신규 방문 고객인가, 기존 고객인가?

---

#### 2단계 | 착수 보고 (5월 말)

**현재 계획에 추가할 것들:**

- **아키텍처 변경 내용 명시**
  - 작년 구조의 문제점(gpt-oss 연동, 시스템 프롬프트 제약, RAG 성능 한계)과 올해 변경 사항을 명확히 대비 정리한다.
  - FabriX 포탈 내에서 Qwen 모델 연동 시 제약 사항(토큰 한도, API 응답 속도 등)을 사전에 확인하여 보고서에 반영한다.

- **리스크 관리 계획**
  - 할루시네이션 발생 시 fallback 처리 방안 (예: "정확한 안내를 위해 상담원을 연결해드릴게요")
  - 서비스 중단 기준 (할루시네이션율 X% 초과 시 자동 비활성화 등)
  - 롤백 절차

- **이해관계자 맵핑**
  - AI 거버넌스 심의 위원회 (심의 일정 사전 확보 필수)
  - IT보안팀 (내부망 배포, FabriX 포탈 접근 권한)
  - 법무/컴플라이언스팀
  - 고객서비스팀 (fallback 대응 프로세스 협의)
  - 상품팀 (팩 건강보험 상품 데이터 제공 협조)

- **RAG 데이터 확보 계획**
  - 팩 건강보험 관련 공식 자료 목록 (약관, 상품설명서, FAQ 등) 작성
  - 데이터 수집·정제 담당자 지정 및 일정 포함

---

#### 3단계 | 개발 (6월~8월)

**현재 계획에 추가할 것들:**

- **테스트 데이터셋 선착수 준비 (6월 중)**
  - 팩 건강보험 관련 예상 고객 질문 300~500개 수집
  - 정답 레이블링 (ground truth) 작업 — 상품팀 협조 필요
  - 이 데이터셋이 없으면 9월 CBT에서 또 같은 문제가 반복된다.

- **모니터링 체계 설계 (개발과 병행)**
  - 응답별 품질 자동 측정 파이프라인 (정답률, 응답 거부율, 에스컬레이션율)
  - 이상 응답 감지 시 알림 체계 (Slack 또는 이메일)
  - 대화 로그 저장 구조 (개인정보처리 기준에 맞게)

- **면책 문구 및 고지 UI 설계**
  - 채팅창 상단 또는 첫 메시지에 "AI가 제공하는 정보는 참고용이며 최종 계약 내용은 약관을 기준으로 합니다" 등의 고지 문구 삽입
  - 법무 검토를 거친 문구로 확정 필요

- **에스컬레이션 플로우 개발**
  - Agent가 답변 불가 판단 시 → 상담원 연결 또는 전화 안내로 자연스럽게 전환하는 UX

---

#### 4단계 | 테스트 (9월)

**현재 계획에 추가할 것들:**

- **CBT 설계 개선**
  - 작년 CBT와 달리, 테스트 케이스를 사전에 정답 데이터셋 기반으로 설계한다.
  - 할루시네이션 유형별(보장 내용 오류 / 보험료 오류 / 존재하지 않는 특약 언급 등) 측정 방식을 명확히 한다.
  - 테스트 참여자는 실제 잠재 고객 프로필(40~50대, 건강보험 관심군) 기준으로 선발한다.

- **AI 거버넌스 심의 일정 역산**
  - 심의 준비 자료 완성 시점: 9월 초 필수
  - 심의 신청 → 결과 수령까지 통상 2~4주 소요되므로 **9월 1일 신청 기준으로 계획 수립**을 권장한다.
  - 심의 불통과 시 수정 후 재심의 기간도 일정에 포함해야 한다.

---

#### 5단계 | 오픈 (10월)

**현재 계획에 추가할 것들:**

- **소프트 론칭 전략**
  - 전체 트래픽에 즉시 오픈하지 않고, 특정 유입 경로(예: 이벤트 페이지) 또는 특정 시간대에만 먼저 노출한다.
  - D+3, D+7 품질 점검 후 전체 확장 여부를 결정한다.

- **오픈 후 모니터링 일정**
  - D+1: 할루시네이션율, 에스컬레이션율 즉시 확인
  - D+7: 고객 만족도, 상담 완결률 1차 집계
  - D+30: KPI 전체 리뷰 및 개선 방향 도출

- **롤백 기준 사전 합의**
  - 할루시네이션율이 임계치(예: 10%) 초과 시 자동 비활성화 또는 수동 점검 후 재오픈하는 기준을 팀 내 사전 합의해둔다.

### 전체 일정 보완안

| 시기 | 주요 과제 | 보완 필요 항목 |
|------|-----------|----------------|
| 5월 초 | 기획 재정의 | 할루시네이션 원인 분석, KPI 정의, 법적 검토 착수, 고객 진입 경로 확정 |
| 5월 말 | 착수 보고 | 아키텍처 변경점 명시, 리스크 관리 계획, 이해관계자 맵핑, RAG 데이터 확보 계획 |
| 6월~8월 | 개발 | 테스트 데이터셋 선착수, 모니터링 체계 설계, 면책 문구 확정, 에스컬레이션 플로우 |
| 9월 | 테스트 | CBT 설계 개선, AI 거버넌스 심의 9월 초 신청, 재심의 기간 버퍼 확보 |
| 10월 | 오픈 | 소프트 론칭, D+1/7/30 모니터링, 롤백 기준 사전 합의 |

---

### 핵심 리스크 3가지

| 리스크 | 원인 | 대응 방안 |
|--------|------|-----------|
| 할루시네이션 재발 | RAG 품질, 모델 성능, 프롬프트 구조 복합 문제 | 원인 유형별 분리 측정 후 각각 대응 / Qwen 모델 사전 Bench test 필수 |
| AI 거버넌스 심의 지연 | 심의 준비 부족 또는 심의 불통과 | 9월 초 신청 → 재심의 버퍼 2주 확보 / 작년 심의 결과에서 지적 사항 사전 반영 |
| 법적 리스크 | 금소법·보험업법 상 AI 설명 범위 불명확 | 5월 중 법무 검토 선착수 → 6월 개발에 반영 |
